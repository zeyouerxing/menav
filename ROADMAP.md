# MeNav 路线图

本文档维护 MeNav 的中期方向、近期执行重点与明确非目标。

它同时服务维护者和使用者

> - 维护者：判断下一步该做什么、不该做什么，以及哪些方向需要单独立项。
> - 使用者：理解 MeNav 当前定位、支持边界和后续演进方向。

本文档不是 changelog，不记录已完成细节；也不是 backlog，不罗列零散任务。
它定位是仓库级执行路线图

## 当前基线

MeNav 当前基线已经明确：

- MeNav 是一个以 `YAML + Markdown + Astro` 为核心的静态导航站生成器。
- 单页模型继续保留：所有导航页面构建到同一个 `index.html`，运行时通过 `?page=<id>` 切换。
- 配置已收敛为 `ResolvedConfig.pages` 和 `buildSiteModel()` 单一站点模型；页面必须由 `navigation.id` 声明，隐藏页使用 `hidden: true`。
- 项目已完成 Astro 现代化迁移，并具备完整的 `lint / test / build / browser contract / audit` 校验链路。
- 浏览器扩展只保留“书签导入入口”这一条链路，不再承担页面运行时 CRUD 或后台职责。
- 如果未来需要后台能力，方向暂定 `Cloudflare Worker + D1 + Access`。

当前判断不是继续快速堆功能，而是进入“边界收敛、范本化、稳态演进”阶段。近期架构清理已经把配置加载、站点模型、搜索索引和包根 API 收窄到更明确的边界，后续重点是稳住这些契约。

## 能力边界

### 核心能力：静态站点生成

这是 MeNav 的主路径，也是默认推荐能力。

- 使用 `config/` 下的 YAML 管理导航结构；`navigation[].id` 是页面身份的唯一来源。
- 使用 `content/` 下的 Markdown 管理内容页。
- 使用 Astro 构建为可直接部署的静态站点。
- 适合希望自托管、可版本管理、无后端依赖的用户。

### 增强能力：书签导入

这是当前保留的浏览器相关增强能力。

- 用户在浏览器里管理书签。
- 浏览器书签通过导出 HTML 或由 MarksVault 推送到仓库的 `bookmarks/` 目录。
- `npm run import-bookmarks` 将书签 HTML 转成 `config/user/pages/bookmarks.yml`，并由拆分后的 parser/icons/serializer/writer 模块完成解析、图标推断、序列化和导航补入。
- GitHub Actions 或本地构建重新生成静态站点。

这条链路的目标是降低导入和同步成本，而不是提供在线后台。

### 后续方向：后台

这不是当前主路径，也不是已启动事项，只是明确保留的后续方向。

- 若后续要支持在线编辑、私有备注、排序调整、层级维护、后台管理等能力，应单独建设动态后台能力。
- 技术方向是 `Cloudflare Worker + D1 + Access`。
- 该能力应作为可选增强，而不是直接替代当前静态主路径。
- 前台优先保持静态部署能力，动态数据更适合通过静态快照或受控 API 提供。

## 明确非目标

以下事项在当前阶段默认不作为优先目标：

- 恢复“浏览器扩展驱动页面 DOM 增删改查”的第二条链路。
- 把浏览器扩展继续扩展为后台、数据库或管理系统替代品。
- 在没有明确复用场景前重新设计插件系统或过度扩展点。
- 为了“更像 Astro”而强行放弃当前单页导航模型。
- 在配置体验、架构边界和文档职责尚未收敛前继续快速增加页面模板。
- 为单次需求引入复杂抽象层，或保留没有明确使用场景的兼容分支。

## 执行路线

### Now

当前阶段优先做“边界收敛 + 范本化基础建设”。

#### 对外叙事与内部边界对齐

- 统一 `README.md`、`config/README.md`、`src/README.md`、`ROADMAP.md` 的职责分工。
- 明确 MeNav 的三类能力边界：静态站点生成、书签导入增强、后续动态后台方向。
- 让新用户和维护者都能快速理解：当前支持什么，不支持什么，为什么这样设计。

完成标志：

- 主要文档之间不再互相打架。
- 浏览器扩展的职责表述稳定，不再出现“既像导入器又像后端”的混合叙事。

#### 配置体验与升级体验优化

- 降低 `config/user/` 的使用摩擦。
- 强化错误提示、迁移说明与“完全替换策略”的表达。
- 继续梳理 schema、默认值整理、页面数据映射之间的边界。

完成标志：

- 新用户更容易完成配置。
- 老用户升级时更少依赖隐性知识。

#### layout / runtime / 构建期边界继续收敛

- 继续拆掉职责过重的入口文件。
- 维护 `buildSiteModel()` 作为页面、导航、运行时配置和搜索来源的单一事实来源。
- 保持包根公开 API 收窄：`loadConfig()`、`buildSiteModel()`、`buildSearchIndex()` 和错误类型。
- 明确哪些逻辑属于构建期，哪些属于 Astro 组件层，哪些属于浏览器 runtime。
- 维持运行时最小公开面，避免再次把内部实现扩成外部契约。

完成标志：

- 关键目录职责更清晰。
- 新增功能时更容易判断应该落在哪一层。
- 不再出现配置加载、页面模型、搜索索引各自重复推导页面结构的路径。

#### 默认示例与仓库范本质量提升

- 默认配置不仅要“能跑”，还要体现推荐实践。
- README 示例、默认配置、页面示例尽量讲同一套故事。
- 让 MeNav 更像可以参考的 Astro 静态项目，而不是只为当前仓库服务的成品。

完成标志：

- 新用户复制默认配置后得到的是一套合理样板，而不是仅供演示的杂糅示例。

### Next

在 Now 阶段完成后，下一阶段优先增强静态体验本身，而不是急于扩张产品形态。

#### 静态内容聚合体验完善

- 继续打磨 `articles` 与 `projects` 的缓存、回退、更新时间和失败提示语义。
- 减少时效性数据能力之间的表现差异。
- 让“有缓存”“无缓存”“同步失败”“降级展示”这些状态更统一。

#### 搜索、导航、移动端与可访问性体验提升

- 持续优化搜索结果体验、页面切换体验和移动端交互。
- 补齐空状态、错误状态、弱网场景表现。
- 在不破坏当前架构的前提下继续优化性能和可访问性。

#### 架构说明与决策文档补齐

- 为关键边界补充更清晰的设计取舍说明。
- 把“为什么保留单页模型”“为什么运行时继续存在”“为什么不把后台塞进扩展”这类判断沉淀下来。
- 让维护者能更快解释项目，而不是反复口头同步。

### Later

Later 只放方向，不放默认承诺。

#### 动态后台能力评估

只有当以下条件基本满足时，才正式启动：

- 静态主路径已经足够稳定。
- 文档和配置边界已经清楚，不再频繁调整基础叙事。
- 出现稳定且持续的后台需求，例如：私有备注、在线编辑、排序调整、层级维护。
- 可以接受“新增一套可选后台能力”带来的维护成本，而不是试图把它伪装成现有静态主路径的小补丁。

启动后应遵循：

- 单独写设计文档，不直接塞进当前静态主路径。
- 默认主路径仍是静态站点生成。
- Worker 负责 API，D1 负责主数据，Access 负责鉴权。
- 不把核心书签数据放进 KV，不把浏览器扩展重新当成后端代理。

#### 更明确的能力共存策略

如果后续动态后台启动，需要单独明确：

- 静态站点生成与动态后台能力如何并存。
- 用户如何从静态主路径迁移到动态后台能力。
- 哪些能力在两条路径下保持一致，哪些能力明确只属于动态后台。

## 近期优先级

按当前判断，最近几个版本建议优先关注：

1. 保持 `ROADMAP.md` / `README.md` / `src/README.md` / `config/README.md` 与架构边界同步。
2. 继续改善配置体验与迁移体验，尤其是 `navigation.id`、隐藏页、孤儿页面报错和完全替换策略。
3. 继续清理 layout、runtime、构建期之间的职责泄漏。
4. 默认示例配置和文档样板整理。
5. 为未来动态后台能力预留清晰进入条件，但不提前开工。

## 决策原则

未来新增功能、重构或新能力评估时，默认按以下原则判断：

1. 是否让普通用户更容易上手、修改和升级？
2. 是否让架构边界更清晰，而不是更模糊？
3. 是否保持“静态主路径优先”的默认方向稳定？
4. 是否会显著抬高长期维护成本？如果会，收益是否足够大？
5. 是否符合“仅实现当前需要，不预留无明确需求的扩展点”？
6. 如果新增的是增强能力，是否已经明确它为什么不应该塞回默认主路径？

如果以上问题大多无法正面回答，该事项默认降级、延后或拒绝。

## 维护方式

建议在以下情况更新本文件：

- 当前主路径发生边界变化。
- 近期优先级排序发生明显调整。
- 决定正式启动一个新的能力方向。
- 某个 Later 方向需要进入 Next 或 Now。
- 发现 README、源码说明和路线图之间再次出现叙事漂移。

若只是记录某次已完成改动，应更新 `CHANGELOG.md` 或对应专题文档，而不是把细节堆到本文件。
